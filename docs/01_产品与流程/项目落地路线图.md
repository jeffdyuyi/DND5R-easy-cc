## 3. 分阶段落地方案

### 阶段 0：补齐最小管理文档

目标：

- 明确第一版必须做什么
- 明确第一版先支持哪些资料
- 明确工程验收标准

动作：

1. 新增 MVP 范围与 PRD
2. 新增分步建卡验收标准
3. 新增主持人资源管理需求说明

完成标志：

- 任一功能都能对应到“目标用户、输入、输出、验收标准”

### 阶段 1：建立前端工程骨架

目标：

- 让项目从“文档库”变成“可运行工程”

动作：

1. 初始化 `React + TypeScript + Vite`
2. 建立 `src/components`, `src/features`, `src/domain`, `src/data`, `src/tests`
3. 建立 ESLint、Prettier、Vitest 基础配置
4. 建立文档中定义的类型目录和 mock 数据目录

完成标志：

- 项目可本地运行
- 可以显示空白欢迎页、角色列表页、建卡流程壳子

### 阶段 2：打通数据流水线

目标：

- 把资料从清洗文档转成程序可消费的结构化数据

动作：

1. 为职业、背景、种族、专长、法术、装备定义 JSON schema / TS types
2. 实现 `staging-data` 校验脚本
3. 实现 `build-data` 脚本，把 staging 输出成运行态数据
4. 为每类数据补最小样本和失败样本

完成标志：

- 至少职业、背景、种族、专长四类数据可以稳定产出 app data

### 阶段 3：实现角色构建核心引擎

目标：

- 先把“能算对”做出来，再做“好看”

动作：

1. 实现 `editableState`
2. 实现 `computedState`
3. 实现 grant/effect 解析器
4. 实现熟练、属性、AC、HP、法术、装备的统一计算入口
5. 为典型职业和背景写规则单测

完成标志：

- 给定一份角色选择输入，能产出稳定的最终角色状态

### 阶段 4：实现分步式建卡向导 MVP

目标：

- 让玩家可以完整走通一条创建链路

建议首版步骤：

1. 选择职业
2. 选择背景
3. 选择种族
4. 分配属性
5. 处理专长与熟练项选择
6. 处理装备
7. 预览角色卡

动作：

1. 先做步骤框架和进度管理
2. 每一步只接入最小必需字段
3. 将待选项和未完成项显示在右侧摘要区
4. 支持中断保存与继续编辑

完成标志：

- 用户可以从零完成一名 1 级角色的创建并保存

### 阶段 5：实现角色管理与角色卡

目标：

- 把“创建”扩展到“管理”

动作：

1. 角色列表
2. 角色详情页
3. 重新进入编辑
4. 导出 / 导入 JSON
5. 法术页、装备页、负重页

完成标志：

- 玩家可以创建、查看、修改、删除、导入导出角色

### 阶段 6：实现主持人资源管理工具

目标：

- 让主持人能管理可用资料，而不是只看静态库

动作：

1. 先做只读资源浏览
2. 再做受控编辑
3. 支持标签、来源、启用状态、战役可见性
4. 为第三方/自制内容提供单独入口

完成标志：

- 主持人可以维护可用资源集，并影响玩家端选择范围

### 阶段 7：测试、回归与发布

目标：

- 降低规则回归和脏数据进入生产的概率

动作：

1. 建规则计算单测
2. 建流程集成测试
3. 建关键数据快照
4. 建发布前检查清单

完成标志：

- 每次数据或规则变更都能自动发现主要回归

## 4. 推荐的实施顺序

不要先做完整 UI，再回头补规则引擎。推荐顺序是：

1. 先补 PRD 与验收标准
2. 再建工程骨架
3. 再打通数据流水线
4. 再做规则计算引擎
5. 最后把 UI 向导和 GM 工具接上去

原因很简单：

- 这个项目的真正难点是“数据和规则的一致性”
- UI 只是承载层
- 如果规则口径和数据契约没先锁住，分步式建卡一定会返工
