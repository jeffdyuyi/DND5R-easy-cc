# 分步建卡验收标准

## 1. 文档目的

本文件定义 MVP 分步建卡向导的步骤范围、每一步的输入输出、校验规则和完成标准。

## 2. 总体验收原则

### 2.1 向导必须满足

- 每一步有明确输入和输出
- 未完成时不能误判为已完成
- 可以返回上一步修改
- 修改上游步骤后，下游结果能够重算
- 所有关键待选项都有明显提示

### 2.2 角色状态必须满足

- 使用 `editableState` 保存建卡过程
- 使用 `computedState` 保存当前计算快照
- 每次关键选择变更后，相关结果可以重新计算

### 2.3 UI 必须满足

- 当前步骤清晰可见
- 已完成、进行中、待完成、存在错误四种状态可区分
- 右侧或底部必须展示当前角色摘要和未完成项

---

## 3. MVP 步骤定义

采用 9 步流程：

1. 种族 (Species)
2. 背景 (Background)
3. 职业 (Class / Level)
4. 法术 (Spells)
5. 属性 (Abilities)
6. 技能 (Skills)
7. 装备 (Equipment)
8. 细节 (Details)
9. 阵营与总结 (Identity & Preview)

---

## 4. 步骤验收标准

### 第 1 步：选择种族

#### 输入

- 种族列表
- 种族授予的特性、速度、体型、感官、语言或法术

#### 用户动作

- 选择 1 个种族
- 完成该种族要求的必选项

#### 系统输出

- `editableState` 记录种族及必选项
- `computedState` 至少更新：
  - 速度
  - 体型
  - 感官
  - 种族特性
  - 种族授予法术或语言

#### 校验规则

- 未选种族时不能进入下一步
- 存在种族待选项未完成时不能进入下一步

#### 完成标准

- 种族及其必选项完成
- 可计算的种族结果已进入统一计算

### 第 2 步：选择背景

#### 输入

- 背景列表
- 背景授予的属性提升、起源专长、熟练、语言、装备分支

#### 用户动作

- 选择 1 个背景
- 完成背景相关必选项

#### 系统输出

- `editableState` 记录背景及背景待选项
- `computedState` 至少更新：
  - 背景属性提升
  - 起源专长授予
  - 背景熟练
  - 背景装备或金币

#### 校验规则

- 未选背景时不能进入下一步
- 背景必选项未完成时不能进入下一步

#### 完成标准

- 背景及其必选项已完整记录
- 背景所引发的一级授予已进入待计算/已计算状态

### 第 3 步：选择职业

#### 输入

- 职业列表
- 可选起始等级范围
- 职业起始装备选项

#### 用户动作

- 选择 1 个职业
- 选择起始等级
- 如果职业提供起始装备分支，则选择其中 1 条

#### 系统输出

- `editableState` 记录职业、等级和职业装备选择
- `computedState` 至少更新：
  - 生命骰基础信息
  - 职业熟练
  - 职业固定授予
  - 是否需要后续插入子职业或额外待选项

#### 校验规则

- 未选职业时不能进入下一步
- 存在职业必选分支未选时不能进入下一步

#### 完成标准

- 用户已选定职业
- 所有职业必需选择已记录
- 当前步骤无未解决阻塞项

### 第 4 步：分配属性

#### 输入

- 属性生成模式
- 六项属性初始值
- 背景或其他来源提供的属性提升

#### 用户动作

- 选择属性生成方式
- 完成六维属性分配

#### 系统输出

- `editableState` 记录基础属性分配方式和结果
- `computedState` 至少更新：
  - 六维属性值
  - 六维调整值
  - 熟练加值
  - 基础负重能力

#### 校验规则

- 六维属性未完整分配时不能进入下一步
- 非法分配方案必须阻止继续

#### 完成标准

- 六维属性齐全且合法
- 基础属性派生值已生成

### 第 5 步：处理专长、熟练和语言等待选项

#### 输入

- 职业、背景、种族、专长授予的待选项
- 技能、工具、语言、法术、专长选择池

#### 用户动作

- 完成所有当前必须完成的待选项

#### 系统输出

- `editableState` 记录每一项选择结果
- `computedState` 至少更新：
  - 技能熟练
  - 工具熟练
  - 武器熟练
  - 护甲训练
  - 已获专长
  - 已获法术
  - 已知语言

#### 校验规则

- 必选项数量不足不能通过
- 同一来源禁止重复选择同一条目
- 若规则要求重选，必须明确提示

#### 完成标准

- 所有阻塞性待选项已完成
- 当前没有未处理的关键授予链

### 第 6 步：处理装备

#### 输入

- 职业装备
- 背景装备
- 金币方案
- 基础物品库

#### 用户动作

- 确认起始装备方案
- 如采用金币方案，完成购买

#### 系统输出

- `editableState` 记录装备选择和购买结果
- `computedState` 至少更新：
  - 装备清单
  - 钱币
  - 负重
  - 负重状态
  - AC 所需装备构成

#### 校验规则

- 必选装备方案未确认不能进入下一步
- 金币不足时不能完成购买

#### 完成标准

- 起始装备来源明确
- 负重与钱币结果可计算

### 第 7 步：预览与保存

#### 输入

- 之前所有步骤的 `editableState`
- 当前统一计算结果

#### 用户动作

- 查看角色摘要
- 可返回任一步修改
- 点击保存

#### 系统输出

- 生成最终角色快照
- 保存角色档案
- 返回角色列表或角色详情

#### 校验规则

- 若仍存在阻塞性未完成项，不允许保存为 playable
- 若仅存在非阻塞文本信息缺失，可保存为 draft

#### 完成标准

- 角色可保存
- 保存后再次打开结果一致

## 5. 跨步骤联动验收

### 5.1 回退重算

验收要求：

- 修改职业后，受影响的熟练、专长、装备、法术结果必须刷新
- 修改背景后，背景来源属性提升和起源专长必须刷新
- 修改种族后，体型、速度、感官和种族法术必须刷新
- 修改属性后，调整值、AC、技能加值等派生结果必须刷新

### 5.2 存档恢复

验收要求：

- 用户中途退出后再次进入，可以恢复到上次编辑状态
- 恢复后当前步骤、已选项、未完成项都要一致

### 5.3 待选项一致性

验收要求：

- 待选项必须来自规则计算，不允许 UI 自行硬编码一套独立逻辑
- 同一选择结果必须能在摘要区和最终角色卡中保持一致

---

## 6. 可接受的首版限制

MVP 可以接受：

- 仅支持单职业 1 级角色完整走通
- 旧版资料只做有限兼容
- 一些纯文本特性只展示，不参与自动计算

MVP 不可接受：

- 角色保存后无法重新编辑
- 关键结果依赖手工回填
- 上游修改后下游结果不刷新
- 必选项漏选却能保存为 playable

---

## 7. 测试样板建议

至少准备以下验收样例：

1. 1 个纯战士型角色
2. 1 个有起源专长授予链的角色
3. 1 个有种族法术授予的角色
4. 1 个采用金币购置装备的角色

这些样例应作为后续集成测试和回归测试的首批固定样本。
