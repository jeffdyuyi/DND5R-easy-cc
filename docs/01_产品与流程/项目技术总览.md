## 1. 项目总览

本文件描述的是一个已经存在过的 D&D 5e (2024) 角色构建器参考实现，用来说明推荐的前端分层、模块拆分和交互形态。

它的用途是：

- 作为后续正式项目的架构参考
- 作为产品与工程讨论时的界面和模块样板
- 帮助把数据、规则和 UI 串成一个可执行的整体
- 作为当前仓库未来工程初始化时的参考命名方案

它不是当前仓库已落地代码的真实状态说明。当前仓库仍以数据和文档为主，真正的前端工程需要按 [项目落地路线图](E:/YJF/DND2024characterbuilder-2.0/docs/01_产品与流程/项目落地路线图.md) 另行初始化。

当前项目仓库地址：

- [jeffdyuyi/DND2024characterbuilder-2.0](https://github.com/jeffdyuyi/DND2024characterbuilder-2.0)

参考实现中的产品形态分为 **玩家端** 和 **主持人端** 两个独立视图。

- **玩家端**：9 步分步建卡向导 + 角色卡库管理 + 法术书
- **主持人端**：11 个数据库模块的管理面板
- **无房间功能**：当前版本不包含联机/房间功能

## 3.5 数据约束与自动统合

为保证角色创建器能够稳定按步骤调取职业、背景、种族、专长、法术与装备数据，并对规则结果进行自动统合，数据层必须遵守统一的数据约束规范。

规范来源：

- [staging-data统一约束规范.md](E:/YJF/DND2024characterbuilder-2.0/docs/02_数据工程/staging-data统一约束规范.md)

### 3.5.1 数据分层

角色创建器运行时依赖三层数据：

- 原始库存层：`中文data`
- 候选清洗层：`staging-data`
- 现有消费层：`data`

其中：

- `中文data` 用于追溯原始规则来源
- `staging-data` 用于统一跨类别结构与 review 结论
- `data` 用于现有前端稳定消费

### 3.5.2 角色构建的最小自动统合目标

角色创建器至少需要把以下结果自动汇总到角色状态中：

- 熟练项
- 护甲等级
- 法术获取
- 专长获取
- 属性调整
- 装备带来的结算影响

### 3.5.3 关键自动统合输入

所有被角色创建器正式消费的数据，必须最终能支持提取以下信息：

- `id`
- `name`
- `source`
- 分类字段
- 可追踪的授予效果
- 可追踪的选择逻辑
- 可追踪的规则修正

### 3.5.4 嵌套授予原则

角色创建流程中允许出现嵌套授予：

- 职业授予专长或法术
- 背景授予专长
- 种族授予法术
- 专长再次授予法术、属性、熟练项或其他选择

因此计算层必须支持“递归式统合”，而不能只处理第一层选择结果。

### 3.5.5 版本与来源原则

- 同名不同 `source` 条目必须并存
- `source` 保留原始缩写，不得粗化替代
- 只要规则效果不同，就不合并，只并存

### 3.5.6 计算层职责

建议将以下逻辑集中到统一计算入口，而不是分散在各 Step 组件中：

- 熟练项去重与来源追踪
- AC 公式计算与修正叠加
- 法术来源合并
- 专长来源合并
- 属性变化应用

这部分应由角色构建计算模块统一处理，UI 只负责选择与展示。

### 3.5.7 计算口径规范的正式依赖

角色创建器的规则计算必须以以下文档为正式口径来源：

- [角色构建计算口径规范.md](E:/YJF/DND2024characterbuilder-2.0/docs/03_规则与状态/角色构建计算口径规范.md)

该文档负责定义：

- 角色构建的阶段性计算顺序
- 熟练、属性、专长、法术、AC 的统一统合口径
- 嵌套授予的递归处理原则
- 装备重量、钱币重量、载重和负重状态的计算规则

本技术文档中的 UI、Context、步骤流和计算模块设计，都必须与该文档保持一致。

### 3.5.8 最终角色状态必须可承接的结果

无论角色创建过程中的中间状态如何组织，最终角色状态都必须能稳定输出以下结果：

- 基础身份：职业、子职业、种族、背景、专长、法术、装备
- 属性结果：属性值、调整值、熟练加值
- 熟练结果：豁免、技能、工具、武器、护甲训练
- 防御结果：最终 AC 与 AC 构成说明
- 施法结果：已知法术、授予法术、戏法、始终准备法术、施法属性来源
- 负重结果：总负重、钱币重量、总负重上限、推拉抬极限、负载比例、负重状态

如果某个已选来源的数据无法汇总到上述结果之一，则该来源数据不能视为“已被角色创建器正式支持”。

### 3.5.9 嵌套授予的消费要求

角色创建流程不能只消费第一层选择结果，必须支持以下嵌套场景：

- 职业授予法术或专长
- 背景授予专长
- 种族授予法术
- 专长再次授予法术、属性、熟练项或其他选择

因此，计算模块必须在每次选择确认后重新执行统一统合，而不是只更新局部展示字段。

最低要求：

- 已获得条目可以再次触发新的待选池
- 新的待选池完成选择后可以继续触发下一层授予
- 同一条目不会因递归处理被重复结算

### 3.5.10 负重与财富的正式产品要求

角色创建器必须将负重系统视为正式显示结果，而不是附加说明。

最低产品要求：

- 显示正常载重上限
- 显示推、拉、抬极限
- 显示当前总负重
- 显示钱币重量
- 显示负载比例
- 显示当前负重状态

若启用变体负重规则，还必须显示：

- 轻度负重阈值
- 重度负重阈值
- 速度惩罚
- 是否触发力量、敏捷、体质相关检定/攻击/豁免劣势

### 3.5.11 模块职责补充

建议在角色构建计算模块中明确拆分以下职责：

- `computeAbilityState`
- `computeProficiencies`
- `computeFeatState`
- `computeSpellState`
- `computeArmorClass`
- `computeCarryingState`
- `computeFinalCharacterState`

其中：

- Step 组件只负责收集输入和驱动选择
- Character Context 负责保存当前选择状态与最终结果
- 统一计算模块负责生成最终可展示、可保存、可复算的角色状态

### 3.5.12 与数据候选层的闭环关系

本项目的数据治理和角色创建计算形成如下闭环：

1. `中文data` 保留原始规则来源
2. `staging-data` 按统一约束整理候选条目
3. `data` 提供当前前端稳定消费的目标形态
4. 角色创建器按照 [角色构建计算口径规范.md](E:/YJF/DND2024characterbuilder-2.0/docs/03_规则与状态/角色构建计算口径规范.md) 对已消费数据执行统一统合
5. 若某类数据无法被角色创建器稳定统合，则回到 `staging-data` 约束层补字段或补 review 结论，而不是在 UI 层临时硬编码规则

这条闭环原则应作为后续扩展职业、第三方资料、魔法物品、复杂专长和复杂种族特性的统一处理方式。

对于当前项目的 MVP，请以 [MVP范围与PRD](E:/YJF/DND2024characterbuilder-2.0/docs/01_产品与流程/MVP范围与PRD.md) 为准，即主持人端首版只要求“浏览、筛选、查看详情”，不要求完整编辑和审批流。

### 5.5 数据持久化机制 (useLibraryManager)

**核心 Hook**: `useLibraryManager.ts`

```typescript
function useLibraryManager<T>(key: string, initialItems: T[]): LibraryHandler<T>
```

- 使用 `useLocalStorage(key, initialItems)` 持久化
- **首次加载**: 从 `data/*.ts` 静态数据初始化到 localStorage
- **后续加载**: 直接从 localStorage 读取
- **CRUD 操作**: `onAdd` / `onUpdate` / `onDelete` / `onImport` → 自动同步到 localStorage
- **导入去重**: 基于 `id` 字段，已存在的条目自动跳过

**数据初始化链路**:
```
data/index.ts
  → 聚合 CLASS_DB, SUBCLASS_DB, SPECIES_DB, ...
    → LibraryContext 中构造 11 个 useLibraryManager 实例
      → 各实例读取 localStorage; 若为空则填充静态数据
        → 对外暴露 { items, onAdd, onUpdate, onDelete, onImport }
```

## 6. 数据类型定义速查

> 当前项目请以 [领域模型与类型契约](E:/YJF/DND2024characterbuilder-2.0/docs/03_规则与状态/领域模型与类型契约.md) 为正式类型口径；`types.ts` 仅作为参考实现中的历史文件名。

### 核心接口

| 接口 | 关键字段 | 用途 |
|------|---------|------|
| `BaseLibraryItem` | `id`, `name`, `description`, `source`, `tags?` | 所有资料库条目的基类 |
| `ClassItem` | extends Base + `hitDie`, `primaryAbility`, `saves`, `coreTraits`, `features[]`, `subclasses[]`, `subclassLevel`, `classTable?`, `spellList?` | 职业 |
| `SubclassItem` | extends Base + `parentClass`, `features[]`, `spellList?` | 子职业 |
| `SpeciesItem` | extends Base + `speed`, `size`, `darkvision`, `traits[]`, `subraces?` | 种族 |
| `BackgroundItem` | extends Base + `abilityScores[]`, `feat`, `skills[]`, `tool`, `equipment[]`, `choices?` | 背景 |
| `SpellItem` | extends Base + `level`, `school`, `castingTime`, `range`, `components`, `duration`, `classes?[]` | 法术 |
| `FeatItem` | extends Base + `category`, `prerequisite?`, `benefits[]` | 专长 |
| `ItemItem` | extends Base + `type`, `cost`, `weight`, `rarity?`, `damage?`, `ac?`, `properties?[]`, `mastery?` | 物品 |

### ContentSource 类型

```typescript
type ContentSource = '官方规则' | '第三方/原创' | string;
```

规则: `source === '官方规则'` 的条目 **只读**，`source === '第三方/原创'` 的条目可 CRUD。
