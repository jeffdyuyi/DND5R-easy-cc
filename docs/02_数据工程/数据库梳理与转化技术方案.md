
# DND5R 角色卡生成器数据库梳理与转化技术方案（细化版）

## 目标
全面梳理、转化和标准化现有数据库文件，确保所有数据结构、字段、内容、引用、分类、格式、初始化、导入导出等均能无歧义、无遗漏地满足角色卡生成器的全部功能需求。

---

## 1. 数据类型与字段标准化

### 1.1 明确所有核心数据类型
每一类数据都必须单独梳理，分别建立标准字段：

#### 1.1.1 职业（ClassItem）
- id（string，唯一，推荐格式 class-xxx）
- name（string，职业中文名）
- name_en（string，职业英文名）
- description（string，完整描述）
- source（string，内容来源）
- tags（string[]，标签）
- hitDie（number，生命骰）
- primaryAbility（string，主属性）
- saves（string[]，豁免）
- coreTraits（string[]，核心特性名）
- features（FeatureItem[]，详见下）
- subclasses（string[]，子职业 id 列表）
- subclassLevel（number，获得子职业的等级）
- classTable（ClassTableRow[]，详见下）
- spellList（string[]，可选，法术列表 id）

#### 1.1.2 子职业（SubclassItem）
- id（string，唯一，subclass-xxx）
- name
- name_en
- description
- source
- tags
- parentClass（string，父职业 id）
- features（FeatureItem[]）
- spellList（string[]，可选）

#### 1.1.3 种族（SpeciesItem）
- id
- name
- name_en
- description
- source
- tags
- speed（number，速度）
- size（string，体型）
- darkvision（number，黑暗视觉距离）
- traits（TraitItem[]，详见下）
- subraces（string[]，可选，子种族 id）

#### 1.1.4 背景（BackgroundItem）
- id
- name
- name_en
- description
- source
- tags
- abilityScores（string[]，能力加值）
- feat（string，专长 id）
- skills（string[]，技能）
- tool（string[]，工具）
- equipment（string[]，起始装备）
- choices（BackgroundChoiceItem[]，可选）

#### 1.1.5 法术（SpellItem）
- id
- name
- name_en
- description
- source
- tags
- level（number，环阶）
- school（string，学派）
- castingTime（string，施法时间）
- range（string，距离）
- components（string[]，VSM）
- duration（string，持续时间）
- classes（string[]，可用职业 id）

#### 1.1.6 专长（FeatItem）
- id
- name
- name_en
- description
- source
- tags
- category（string，专长类别）
- prerequisite（string，前置条件）
- benefits（string[]，效果描述）

#### 1.1.7 物品（ItemItem）
- id
- name
- name_en
- description
- source
- tags
- type（string，物品类型：武器/护甲/工具/冒险物品/魔法物品）
- cost（string，价格）
- weight（number，重量）
- rarity（string，稀有度，魔法物品必填）
- damage（string，伤害，武器必填）
- ac（number，护甲等级，护甲必填）
- properties（string[]，属性/特性）
- mastery（string[]，精通类型）

#### 1.1.8 其他嵌套类型
- FeatureItem：id, name, description, level, source, tags
- TraitItem：id, name, description, source, tags
- ClassTableRow：level, proficiencyBonus, features, spellSlots, ...
- BackgroundChoiceItem：type, options, count, description

### 1.2 字段要求
- 所有字段必须完整，不能省略。无内容时填空字符串、空数组或 null。
- 字段命名、类型、嵌套结构必须与 types.ts 保持一致。

## 2. 数据内容清洗与去重

### 2.1 唯一性校验
- 每条数据的 id 必须唯一，且全局无重复。
- id 命名规范：class-xxx、subclass-xxx、race-xxx、spell-xxx、feat-xxx、item-xxx。

### 2.2 冗余字段清理
- 删除所有无关、重复、无效字段。
- 仅保留前端和业务实际需要的字段。

### 2.3 内容完整性检查
- 检查所有必填字段是否有值。
- 缺失内容需补全或填空。

---

## 3. 关联关系梳理

### 3.1 明确所有引用关系
- 子职业的 parentClass 必须指向有效职业 id。
- 法术、专长、物品等的 classes、category、type 字段必须与其他数据类型 id 对应。
- 背景、专长等的 prerequisite 字段需与相关 id 或条件文本一致。
- 物品的 type 字段必须为：weapon、armor、tool、gear、magic。

### 3.2 关联信息补全
- 若原始数据未包含必要引用，需手动或脚本补全。
- 所有引用均需校验目标 id 是否存在。

## 4. 数据分层与分类

### 4.3 现有消费参照层

- [data](E:/YJF/DND2024characterbuilder-2.0/data)
- 作用：提供当前前端已验证可读的目标字段形态参考
- 原则：本轮视为只读基准，不直接被未审查数据污染

### 4.4 数据发布层（新增）

- [data-builder](E:/YJF/DND2024characterbuilder-2.0/data-builder)
- 作用：存放清洗转换后的 TS 消费层数据，用于角色创建器正式调取
- 原则：绝对禁止覆盖原始 `data` 目录；所有 .ts 文件由脚本从 `staging-data` 转换生成，确保类型绑定和全库一致性。

**重要强制要求：所有描述性文字（如 name、description、traits、features 及所有嵌套字段中的文本内容）必须与中文data原始内容完全一致，不得进行任何翻译、缩写、扩写或修改，必须保持原文原貌，严禁自行增删改动。**

---

## 5. 字段补全与格式转化

### 5.1 字段补全
- 所有前端所需字段必须补全。
- 缺失字段填空字符串、空数组或 null，不能省略。
- 复杂嵌套（如 features、traits、classTable）必须结构化为数组或对象。

### 5.2 格式统一
- Markdown、非标准 JSON、表格等全部转为标准 JSON 或 TypeScript 对象。
- 字段命名、类型、嵌套结构与 types.ts 完全一致。

### 5.3 数据示例
详见附录。

---

## 6. 本地化与多语言支持

### 6.1 多语言字段
- 每条数据必须有 name_en、description_en 字段。
- 若有多语言需求，可扩展为 { zh: string, en: string } 结构。

---

## 7. 初始化与导入导出机制

### 7.1 静态数据初始化
- 所有标准化数据以静态 JSON/TS 文件形式存储于 `data/` 或 `data-builder/` 目录。
- 首次加载时自动导入 localStorage。

### 7.2 导入导出
- 支持批量 JSON 导入导出。
- 导入时自动去重（以 id 为准），校验字段完整性。
- 导出时字段齐全，结构标准。

### 7.3 数据发布脚本
- 所有由 `staging-data` 候选条目推送到 `data-builder` 消费层的内容，必须通过自动化转换脚本（如 `scripts/publish_xxx_ts.js`）完成，禁止手动修改。
