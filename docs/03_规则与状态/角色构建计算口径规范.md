# 角色构建计算口径规范

## 1. 文档目标

本规范定义角色创建器在“分步骤选择数据并自动统合规则结果”时所采用的统一计算口径。  
它不负责规定原始数据怎么清洗，而是负责规定：

- 角色创建流程中哪些数据必须进入计算
- 同类效果来自多个来源时如何统合
- 嵌套授予如何递归处理
- 哪些结果必须写回最终角色状态

本规范用于约束：

- 产品侧的交互与展示逻辑
- 程序侧的计算模块与状态结构
- 数据侧的候选条目是否足以支撑自动计算

## 2. 适用范围

本规范适用于角色创建流程中以下主模块：

- 职业
- 子职业
- 种族
- 背景
- 专长
- 法术
- 装备
- 魔法物品

本规范面向的核心角色对象至少要能承接：

- 选择结果
- 汇总后的结算结果
- 各结算结果的来源链

## 3. 计算层总原则

### 3.1 选择与计算分离

UI 组件负责：

- 呈现可选项
- 记录用户选择
- 展示当前阶段的局部结果

计算层负责：

- 将所有来源的数据效果统一汇总
- 处理固定授予与可选授予
- 解析嵌套授予
- 生成最终角色状态

### 3.2 文本与结算分离

- 文本用于展示和审阅
- 结算必须依赖结构化结果或可稳定抽取的统一字段

若某个效果只存在于长文本中，且无法稳定抽取，则该条目不得视为“可自动计算完成”。

### 3.3 来源可追踪

所有自动计算结果都应尽量保留来源链。  
例如熟练项、法术、专长、AC 修正都应至少知道来自：

- 职业
- 背景
- 种族
- 专长
- 物品
- 魔法物品

### 3.4 递归统合

角色创建器必须支持嵌套授予。

示例：

- 背景授予专长
- 专长再授予法术
- 法术或特性又要求选择施法属性
- 种族授予法术列表
- 职业或子职业授予额外专长选择

因此计算应采用“递归收集 -> 去重 -> 结算输出”的方式，而不是只处理第一层。

## 4. 角色创建流程中的计算阶段

建议将角色创建过程拆成以下 6 个计算阶段：

### 阶段 1：基础身份建立

输入：

- 职业
- 子职业
- 种族
- 背景
- 基础等级

输出：

- 角色基础元数据
- 可用选择池
- 初始固定授予

### 阶段 2：属性与基础派生建立

输入：

- 初始属性值
- 种族/背景/专长的属性提升

输出：

- 最终属性值
- 属性调整值
- 负重基础值

### 阶段 3：熟练与训练统合

输入：

- 职业熟练
- 背景熟练
- 种族熟练
- 专长熟练
- 其他特性授予

输出：

- 豁免熟练
- 技能熟练
- 工具熟练
- 武器熟练
- 护甲训练

### 阶段 4：施法与专长统合

输入：

- 职业施法
- 种族法术
- 背景授予法术/专长
- 专长授予法术/专长

输出：

- 已获专长
- 已获法术
- 可选法术池
- 施法属性来源

### 阶段 5：装备与护甲等级统合

输入：

- 装备
- 护甲
- 盾牌
- AC 修正特性
- 装备重量
- 钱币重量

输出：

- 最终 AC
- AC 构成说明
- 当前负重
- 负重状态

### 阶段 6：最终角色状态汇总

输出：

- 角色可展示状态
- 角色可保存状态
- 来源追踪信息

## 6. 熟练项统合口径

### 6.1 统合范围

熟练相关结果必须至少区分为以下五类：

- 豁免熟练
- 技能熟练
- 工具熟练
- 武器熟练
- 护甲训练

### 6.2 来源优先级

熟练项通常不是覆盖关系，而是并集关系。  
统一口径为：

- 所有来源先全部收集
- 相同结果去重
- 同一项允许保留多个来源

### 6.3 去重原则

若两个来源都授予同一项熟练：

- 结果中只保留一份熟练项
- 来源记录中保留多个来源

### 6.4 选择型熟练

若某来源授予“从若干项中选择”：

- 未做选择前，该内容进入待选池
- 完成选择后，才并入正式熟练结果

### 6.5 自动统合要求

以下情形必须正确统合：

- 职业给定技能熟练
- 背景额外给技能熟练
- 种族给工具熟练
- 专长再次给技能或工具熟练
- 已有熟练被重复授予

---

## 7. 属性统合口径

### 7.1 统合来源

属性调整可能来自：

- 背景
- 种族
- 专长
- 其他授予效果

### 7.2 应用顺序

建议顺序：

1. 基础属性输入
2. 固定属性提升
3. 选择型属性提升
4. 后续来源的属性修正
5. 最大值限制检查

### 7.3 最大值限制

若某条目说明属性值提升但不能超过某上限，则：

- 正常提升后
- 再应用上限裁剪

### 7.4 记录要求

每次属性变化都应保留：

- 来源
- 变化值
- 是否固定/可选
- 是否受上限约束

---

## 8. 专长统合口径

### 8.1 专长来源

专长可能来自：

- 用户主动选择
- 背景授予
- 职业授予
- 子职业授予
- 其他专长再授予

### 8.2 统合规则

- 同一专长以 `id` 为唯一键
- 同一 `id` 重复授予时不重复加入
- 但必须记录多个来源

### 8.3 前置条件

前置条件检查必须基于：

- 等级
- 属性
- 施法能力
- 种族
- 背景
- 已拥有专长
- 已拥有特性

### 8.4 可重复专长

若专长 `repeatable = true`：

- 允许多次选择
- 但必须记录每次选择的来源和附带选择内容

若 `repeatable` 未明确标记：

- 默认不可重复

### 8.5 嵌套授予

若专长会再授予：

- 法术
- 属性提升
- 熟练项
- 其他选择

则必须继续进入递归统合流程，直到没有新的可结算结果。

---

## 9. 法术统合口径

### 9.1 法术来源

法术可能来自：

- 职业
- 子职业
- 种族
- 背景
- 专长
- 魔法物品

### 9.2 法术分类

法术必须至少区分：

- 戏法
- 普通已知法术
- 始终准备法术
- 固定授予法术
- 免费施放法术
- 每长休/短休可用法术

### 9.3 去重规则

若相同法术被多个来源授予：

- 主结果中只保留一个法术条目
- 但需记录所有来源
- 若授予方式不同，需保留授予方式差异

示例：

- 一个来源“始终准备”
- 另一个来源“每日免费施放一次”

则这两个效果不能互相覆盖。

### 9.4 施法属性

若法术来源会指定施法属性：

- 必须记录施法属性来源
- 同名法术来自不同来源时，其施法属性可以不同

### 9.5 选择型法术

若某来源只授予“可从某列表中选法术”：

- 在用户未完成选择前，进入待选池
- 完成选择后，才进入最终法术列表

---

## 10. 护甲等级统合口径

### 10.1 AC 来源

AC 可能来自：

- 无甲基础
- 护甲
- 盾牌
- 职业特性
- 种族特性
- 专长修正
- 魔法物品修正

### 10.2 统一计算顺序

建议按以下顺序计算：

1. 确定当前采用的基础 AC 公式
2. 应用护甲公式
3. 应用盾牌加值
4. 应用职业/种族/专长修正
5. 应用魔法修正
6. 生成最终 AC 与构成说明

### 10.3 基础 AC 公式

至少需要支持：

- `10 + 敏捷调整值`
- 护甲给定固定 AC
- 护甲给定 `基础 + 敏捷修正`
- 中甲类 `基础 + 敏捷修正(上限2)`
- 盾牌额外 `+2` 或更高

### 10.4 特性型 AC

若职业或种族提供替代 AC 公式，例如：

- 无甲防御
- 天生护甲

则应视为“候选基础公式”，最终由角色当前实际装备状态决定采用哪一个。

### 10.5 输出要求

最终必须输出：

- `armorClass`
- `armorClassBreakdown`

其中 `armorClassBreakdown` 应至少能解释：

- 基础公式
- 护甲来源
- 盾牌来源
- 额外修正来源

## 11. 负重、抬举、推拉与财富重量口径

本节为强制纳入角色创建器的统一计算规则。

### 11.1 基础负重规则

依据力量值计算：

- `carryCapacityLb = 力量值 × 15`

说明：

- 这是正常载重上限
- 单位为磅（lb）

### 11.2 推、拉、抬上限

依据力量值计算：

- `pushDragLiftLb = 力量值 × 30`

说明：

- 即双倍载重
- 当推拉重量高于正常载重时，角色速度减为 `5 尺`

### 11.3 体型修正

以中型为基准：

- 中型：基准倍率 `1`
- 大型：倍率 `2`
- 巨型：倍率 `4`
- 超巨型：倍率 `8`
- 小型：倍率 `1`
- 微型：倍率 `0.5`

应用规则：

- `carryCapacityLb = 力量值 × 15 × sizeMultiplier`
- `pushDragLiftLb = 力量值 × 30 × sizeMultiplier`

说明：

- 小型与中型按同一载重口径处理
- 微型减半
- 比中型更大的体型每提升一级翻倍

### 11.4 变体：负重

若项目启用 `Variant: Encumbrance`，则在基础负重规则之上再额外计算负重状态。

#### 轻度以上负重阈值

- `encumberedThresholdLb = 力量值 × 5 × sizeMultiplier`

若当前负重 `>` 此值，则角色进入：

- `encumbered`

效果：

- 速度减少 `10 尺`

#### 重度负重阈值

- `heavilyEncumberedThresholdLb = 力量值 × 10 × sizeMultiplier`

若当前负重 `>` 此值，则角色进入：

- `heavily_encumbered`

效果：

- 速度减少 `20 尺`
- 力量、敏捷、体质相关的：
  - 属性检定
  - 攻击检定
  - 豁免检定
  均具有劣势

### 11.5 钱币重量

统一口径：

- `50 枚钱币 = 1 磅`

支持的钱币种类：

- `CP`
- `SP`
- `EP`
- `GP`
- `PP`

钱币总数计算：

- `totalCoins = cp + sp + ep + gp + pp`
- `coinWeightLb = totalCoins / 50`

### 11.6 钱币价值换算

统一金币价值换算：

- `CP = 1/100 GP`
- `SP = 1/10 GP`
- `EP = 1/2 GP`
- `GP = 1 GP`
- `PP = 10 GP`

总金币价值计算：

- `coinValueGp = cp / 100 + sp / 10 + ep / 2 + gp + pp * 10`

### 11.7 当前负重

角色当前负重必须包含：

- 已装备物品重量
- 背包与库存物品重量
- 钱币重量

统一公式：

- `currentCarryWeightLb = equipmentWeightLb + inventoryWeightLb + coinWeightLb`

若项目当前不拆 `equipmentWeightLb` 与 `inventoryWeightLb`，则至少要保证：

- `currentCarryWeightLb = totalItemWeightLb + coinWeightLb`

### 11.8 负载比例

为方便产品展示，角色状态中必须提供负载比例。

统一公式：

- `encumbranceRatio = currentCarryWeightLb / carryCapacityLb`

建议展示为：

- 百分比
- 进度条
- 当前状态标签

示例：

- `12 / 120 lb (10%)`
- `78 / 120 lb (65%)`

### 11.9 负重状态输出

最终状态至少区分：

- `normal`
- `encumbered`
- `heavily_encumbered`
- `over_capacity`

建议判定：

- 若未启用变体规则：
  - `<= carryCapacityLb` -> `normal`
  - `> carryCapacityLb` -> `over_capacity`
- 若启用变体规则：
  - `<= encumberedThresholdLb` -> `normal`
  - `> encumberedThresholdLb` 且 `<= heavilyEncumberedThresholdLb` -> `encumbered`
  - `> heavilyEncumberedThresholdLb` 且 `<= carryCapacityLb` -> `heavily_encumbered`
  - `> carryCapacityLb` -> `over_capacity`

### 11.10 产品展示要求

角色卡或装备页面至少需要显示：

- 正常载重上限
- 推/拉/抬极限
- 当前负重
- 钱币重量
- 负载比例
- 当前负重状态

若启用变体负重，还应显示：

- 轻度负重阈值
- 重度负重阈值
- 当前是否触发速度减值与劣势

## 12. 装备重量计算口径

### 12.1 计入范围

默认计入当前负重的内容：

- 已装备护甲
- 已装备武器
- 背包中物品
- 工具
- 杂物
- 钱币

### 12.2 不计入范围

以下情况应允许通过产品策略配置是否忽略：

- 已消耗的物品
- 纯展示占位物品
- 无重量条目

### 12.3 缺失重量

若条目无重量：

- 不直接阻断角色创建
- 但必须在来源记录中标记 `weight_unknown`
- 默认按 `0` 计入，直到人工补录

## 13. 嵌套授予与递归结算口径

### 13.1 统一递归流程

建议流程：

1. 收集第一层选择结果
2. 提取固定授予
3. 提取可选授予池
4. 应用用户在当前层做出的选择
5. 继续扫描新获得内容是否再授予其他结果
6. 直到没有新增可结算结果

### 13.2 循环保护

递归统合必须防止死循环。

建议规则：

- 同一 `id + grantContext` 只处理一次
- 同一来源重复授予同一结果时只追加来源，不重复结算

### 13.3 结果分类

递归过程中新增结果应至少区分：

- `granted_feat`
- `granted_spell`
- `granted_proficiency`
- `granted_ability_bonus`
- `granted_choice_pool`

## 14. 统一计算输出结构建议

建议在最终角色状态中保留一个专门的“计算结果对象”，例如：

```ts
computed: {
  proficiencies: {
    saves: string[]
    skills: string[]
    tools: string[]
    weapons: string[]
    armors: string[]
  },
  spells: {
    known: string[]
    granted: string[]
    alwaysPrepared: string[]
    cantrips: string[]
  },
  feats: {
    owned: string[]
  },
  defense: {
    armorClass: number
    breakdown: string[]
  },
  carrying: {
    carryCapacityLb: number
    pushDragLiftLb: number
    currentCarryWeightLb: number
    coinWeightLb: number
    encumbranceRatio: number
    encumbranceState: 'normal' | 'encumbered' | 'heavily_encumbered' | 'over_capacity'
  }
}
```

说明：

- 最终字段名可与实际项目不同
- 但必须有等价能力

## 15. 统一校验口径

一个角色构建结果只有满足以下条件，才可视为“结算完成”：

- 所有已选择来源都被纳入计算
- 所有固定授予都已应用
- 所有用户已确认的可选授予都已应用
- 嵌套授予已递归结算完成
- 熟练结果已去重
- AC 已产出最终值和构成说明
- 负重已产出总量、比例与状态
- 关键字段不存在只埋在长文本里的情况

## 17. 执行建议

建议后续实现时，把所有自动统合逻辑集中在统一计算模块中，不要分散在各 Step 组件内部。  
最低建议拆成这些计算入口：

- `computeAbilityState`
- `computeProficiencies`
- `computeFeatState`
- `computeSpellState`
- `computeArmorClass`
- `computeCarryingState`
- `computeFinalCharacterState`
