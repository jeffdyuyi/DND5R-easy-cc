# 领域模型与类型契约

## 1. 文档目标

本文件用于定义项目在工程实现阶段的核心领域模型和 TypeScript 类型边界。

它解决的问题是：

- `staging-data` 中的候选数据如何进入应用
- 应用运行时消费哪些标准模型
- 编辑态、计算态、存档壳之间如何分层
- 哪些字段属于展示，哪些字段属于规则计算，哪些字段只用于追踪来源

## 2. 类型分层

项目中的类型建议分为 5 层：

1. 原始库存层类型
2. 候选清洗层类型
3. 应用消费层领域类型
4. 角色编辑态类型
5. 角色计算态与存档类型

统一方向：

`RawSource -> StagingEntry -> DomainEntity -> EditableCharacterState -> ComputedCharacterState -> CharacterRecord`

说明：

- 原始库存层只负责保留资料，不作为前端主消费类型
- 候选清洗层用于校验、补充 review 和结构归一
- 应用消费层是前端和规则引擎真正依赖的主模型
- 编辑态负责记录用户选择和向导进度
- 计算态负责输出角色最终展示与规则结算快照

## 3. 通用基础类型

### 3.1 标识与来源

```ts
type EntityId = string
type SourceCode = string
type SourceGroup = 'official' | 'partner' | 'third_party' | 'playtest'
type EditionGroup = 'legacy_2014' | 'rules_2024' | 'expansion'
```

### 3.2 状态与审查

```ts
type EntryStatus =
  | 'parsed'
  | 'parsed_with_warnings'
  | 'needs_manual_review'
  | 'blocked'
```

### 3.3 通用规则标签

```ts
type AbilityKey = 'str' | 'dex' | 'con' | 'int' | 'wis' | 'cha'
type SkillKey = string
type LanguageKey = string
type ToolKey = string
type WeaponGroupKey = string
type ArmorGroupKey = string
type SpellSchoolKey = string
type SizeKey = string
type CurrencyKey = 'cp' | 'sp' | 'ep' | 'gp' | 'pp'
```

### 3.4 通用映射类型

```ts
type AbilityScoreMap = Record<AbilityKey, number>
type CurrencyMap = Partial<Record<CurrencyKey, number>>
```

## 4. 候选清洗层类型

候选清洗层保留 review 信息，但已经具备程序可消费的结构。

### 4.1 通用候选条目基类

```ts
type StagingBaseEntry = {
  id: EntityId
  name: string
  nameEn?: string
  source: SourceCode
  sourceGroup?: SourceGroup
  editionGroup?: EditionGroup
  status: EntryStatus
  sourceFile: string
  reviewNotes: string[]
  warnings: string[]
  tags?: string[]
  text?: {
    summary?: string
    description?: string
  }
}
```

### 4.2 规则效果与授予

为避免把规则逻辑散落在不同实体结构里，建议统一定义效果与授予。

```ts
type RuleEffect =
  | AbilityBonusEffect
  | ProficiencyGrantEffect
  | SpellGrantEffect
  | FeatGrantEffect
  | SpeedModifierEffect
  | ArmorClassFormulaEffect
  | ResourceGrantEffect
  | ChoiceGrantEffect

type RuleGrant =
  | FixedGrant
  | ChoiceGrant

type FixedGrant = {
  kind: 'fixed'
  effects: RuleEffect[]
}

type ChoiceGrant = {
  kind: 'choice'
  choiceId: string
  prompt: string
  count: number
  options: ChoiceOption[]
}

type ChoiceOption = {
  id: string
  label: string
  description?: string
  effects?: RuleEffect[]
  grants?: RuleGrant[]
  refs?: EntityReference[]
}
```

### 4.3 引用类型

```ts
type EntityReference = {
  entityType: DomainEntityType
  entityId: EntityId
}

type DomainEntityType =
  | 'class'
  | 'subclass'
  | 'species'
  | 'background'
  | 'feat'
  | 'spell'
  | 'item'
  | 'magic_item'
```

## 5. 应用消费层领域类型

应用消费层类型不再保留复杂 review 细节，只保留运行时需要的稳定字段。

### 5.1 通用实体基类

```ts
type DomainBaseEntity = {
  id: EntityId
  type: DomainEntityType
  name: string
  source: SourceCode
  sourceGroup?: SourceGroup
  editionGroup?: EditionGroup
  tags: string[]
  text?: {
    summary?: string
    description?: string
  }
  grants?: RuleGrant[]
  effects?: RuleEffect[]
}
```

### 5.2 职业与子职业

```ts
type ClassEntity = DomainBaseEntity & {
  type: 'class'
  hitDie: number
  primaryAbilities?: AbilityKey[]
  savingThrowProficiencies: AbilityKey[]
  startingEquipmentOptions?: StartingEquipmentPackage[]
  subclassSelection?: {
    level: number
    subclassIds: EntityId[]
  }
  levelEntries: ClassLevelEntry[]
}

type SubclassEntity = DomainBaseEntity & {
  type: 'subclass'
  parentClassId: EntityId
  levelEntries: ClassLevelEntry[]
}

type ClassLevelEntry = {
  level: number
  proficiencyBonus?: number
  grants?: RuleGrant[]
  effects?: RuleEffect[]
}
```

### 5.3 背景

```ts
type BackgroundEntity = DomainBaseEntity & {
  type: 'background'
  abilityScoreOptions?: AbilityScoreAssignmentRule
  originFeatOptions?: EntityId[]
  skillChoice?: SelectionRule<SkillKey>
  toolChoice?: SelectionRule<ToolKey>
  languageChoice?: SelectionRule<LanguageKey>
  startingEquipmentOptions?: StartingEquipmentPackage[]
}
```

### 5.4 种族

```ts
type SpeciesEntity = DomainBaseEntity & {
  type: 'species'
  size: SizeKey
  speed: MovementProfile
  senses?: SenseProfile[]
  languageChoice?: SelectionRule<LanguageKey>
}
```

### 5.5 专长

```ts
type FeatEntity = DomainBaseEntity & {
  type: 'feat'
  featCategory?: 'origin' | 'general' | 'epic_boon' | 'other'
  prerequisites?: PrerequisiteRule[]
}
```

### 5.6 法术

```ts
type SpellEntity = DomainBaseEntity & {
  type: 'spell'
  level: number
  school: SpellSchoolKey
  castingTime?: string
  range?: string
  duration?: string
  components?: string[]
  classIds?: EntityId[]
}
```

### 5.7 装备与魔法物品

```ts
type ItemEntity = DomainBaseEntity & {
  type: 'item' | 'magic_item'
  itemCategory: 'weapon' | 'armor' | 'tool' | 'gear' | 'consumable' | 'focus' | 'other'
  cost?: CurrencyMap
  weight?: number
  armorData?: ArmorData
  weaponData?: WeaponData
  stackable?: boolean
}

type ArmorData = {
  baseAc: number
  dexCap?: number | null
  allowsShield?: boolean
  armorType?: 'light' | 'medium' | 'heavy' | 'shield' | 'other'
}

type WeaponData = {
  damage?: string
  damageType?: string
  properties?: string[]
  mastery?: string[]
  attackAbility?: AbilityKey | 'auto'
}
```

## 6. 通用辅助规则类型

### 6.1 选择规则

```ts
type SelectionRule<TValue extends string = string> = {
  choose: number
  from: TValue[]
  allowDuplicates?: boolean
}
```

### 6.2 属性分配规则

```ts
type AbilityScoreAssignmentRule = {
  mode: 'fixed_options' | 'choose_any'
  options?: AbilityScoreAssignmentOption[]

  amountPerChoice?: number

type AbilityScoreAssignmentOption = {
  id: string
  grants: Partial<Record<AbilityKey, number>>
}
```

### 6.3 移动与感官

```ts
type MovementProfile = {
  walk: number
  fly?: number
  swim?: number
  climb?: number
  burrow?: number
}

type SenseProfile = {
  type: string
  range?: number
  note?: string
}
```

### 6.4 前置条件

```ts
type PrerequisiteRule =
  | { kind: 'ability_min'; ability: AbilityKey; score: number }
  | { kind: 'level_min'; level: number }
  | { kind: 'class_required'; classId: EntityId }

```

### 6.5 装备包

```ts

  id: string
  label: string
```

## 7. 编辑态类型契约

## 8. 计算态类型契约

计算态是由编辑态与领域数据统一得出的快照。

### 8.1 顶层结构

```ts
type ComputedCharacterState = FinalCharacterState
```

说明：

- 计算态直接复用最终角色状态结构
- 不再单独定义一套平行但相似的展示模型

### 8.2 额外建议

为支持调试与重算，建议在 `computed` 块中保留以下辅助字段：

```ts
type CharacterComputedState = {
  resolvedChoiceIds: string[]
  appliedEffectIds?: string[]
  unresolvedWarnings?: string[]
  calculationVersion?: string
}
```

## 9. 存档壳类型

实际存档不应只存 `editableState` 或 `computedState` 其中之一，而应保存统一记录壳。

```ts
type CharacterRecord = {
  recordVersion: string
  editableState: EditableCharacterState
  computedState: ComputedCharacterState
}
```

约束：

- 打开角色继续编辑时，以 `editableState` 为准
- 展示角色详情时，以 `computedState` 为准
- 任意关键输入变化后，必须重新生成 `computedState`

## 10. 类型边界原则

### 10.1 不允许把规则结算只写在文本里

如果某个条目的规则效果无法进入 `effects` / `grants` / `prerequisites` 这类结构字段，就不能视为可自动计算完成。

### 10.2 不允许让 UI 成为规则来源

UI 只能消费类型契约，不应自行硬编码：

- 某职业给什么熟练
- 某背景给什么专长
- 某种族给什么法术

### 10.3 不允许编辑态和计算态互相污染

- 编辑态不保存最终 AC、最终技能总表、最终法术总表
- 计算态不保存“当前停在哪一步”“未完成待选项”等过程状态

## 11. 实施建议

后续工程中建议按以下文件拆分类型：

```ts
src/domain/types/common.ts
src/domain/types/effects.ts
src/domain/types/entities.ts
src/domain/types/editable-character.ts
src/domain/types/computed-character.ts
src/domain/types/character-record.ts
```

这样拆分的好处：

- 领域实体和角色状态分离
- 规则效果类型可以独立演化
- 更便于后续写计算引擎和数据校验脚本

## 12. 当前结论

当前最重要的契约是：

1. 应用消费层统一以领域实体模型为准
2. 用户输入统一落到 `EditableCharacterState`
3. 规则计算统一输出 `ComputedCharacterState`
4. 最终存档统一使用 `CharacterRecord`
