# 编辑态与计算态角色对象规范

## 1. 文档目标

本文件用于明确角色创建器中两类角色对象的分工与边界：

- 编辑态角色对象
- 计算态角色对象

目标是同时满足以下需求：

- 角色创建过程中可以分步骤编辑
- 角色创建完成后可以重新进入并继续修改
- 角色卡页面可以稳定显示最终结果
- 所有规则结果可以统一重算，而不是靠手工回填

## 2. 核心结论

角色存档不应只保存一种对象，而应同时保存：

- `editableState`
- `computedState`

其中：

- `editableState` 是角色的可编辑真源
- `computedState` 是基于 `editableState` 计算出来的展示与结算快照

统一关系：

`editableState -> 统一计算 -> computedState`

---

## 3. 编辑态角色对象的定义

编辑态角色对象用于保存：

- 用户已经做出的选择
- 尚未完成的选择
- 当前向导步骤状态
- 待选池
- 临时输入

它服务于“继续编辑”而不是“最终展示”。

### 3.1 编辑态的特征

- 可以不完整
- 可以存在待选项
- 可以存在未结算项
- 更接近向导和表单逻辑
- 必须足以恢复创建过程

### 3.2 编辑态必须能回答的问题

- 用户选了什么
- 用户还没选什么
- 当前卡在哪一步
- 哪些选择是固定的，哪些是待确认的
- 哪些嵌套授予已经处理，哪些还没处理

---

## 4. 计算态角色对象的定义

计算态角色对象用于保存：

- 所有选择经过规则计算后的最终结果
- 角色卡页面直接展示的数据
- 各类规则结算结果
- 来源追踪与构成说明

它服务于“展示、调用、导出、复算”。

### 4.1 计算态的特征

- 应尽量完整
- 应能直接渲染角色卡
- 应包含最终数值
- 应包含构成说明
- 应能被重新生成

### 4.2 计算态必须能回答的问题

- 角色最终 AC 是多少
- 角色最终熟练项有哪些
- 角色会哪些法术
- 当前负重与载重状态如何
- 某个结果来自哪几个来源

---

## 5. 为什么必须同时存在

如果只有编辑态：

- 角色卡展示需要实时重新拼装大量内容
- 页面切换和导出会重复结算
- 很难快速得到最终结果

如果只有计算态：

- 用户无法稳定回到创建流程继续修改
- 很多中间选择会丢失
- 无法知道某个最终结果是怎么选出来的

因此正确做法是：

- 编辑时以编辑态为准
- 展示时以计算态为准
- 修改后总是从编辑态重算计算态

---

## 6. 编辑态的职责边界

编辑态角色对象只负责保存“输入”和“过程状态”，不负责保存所有最终规则结果。

### 6.1 编辑态应保存的内容

- 当前职业、子职业、种族、背景选择
- 用户显式选择过的专长
- 用户显式选择过的法术
- 属性录入值
- 技能/工具/语言等待选项的选择结果
- 背包、装备、钱币输入
- 当前步骤进度
- 尚未完成的待选池

### 6.2 编辑态不应承担的内容

- 最终 AC
- 最终熟练总表
- 最终法术汇总表
- 最终负重状态
- 最终所有来源合并结果

这些结果应由计算态承接。

---

## 7. 计算态的职责边界

计算态角色对象只负责保存“基于编辑态算出来的结果”，不负责反向替代编辑态。

### 7.1 计算态应保存的内容

- 最终属性值与调整值
- 最终 AC 与构成说明
- 最终熟练项
- 最终法术结果
- 最终专长结果
- 最终装备与负重结果
- 来源追踪

### 7.2 计算态不应承担的内容

- 向导当前停在第几步
- 用户某一步还没完成的待选项
- 还未确认的法术池
- 尚未提交的专长选择

这些都是编辑态的职责。

## 8. 编辑态角色对象建议结构

```ts
type EditableCharacterState = {
  meta: EditableMeta
  wizard: EditableWizardState
  identity: EditableIdentity
  build: EditableBuildState
  choices: EditableChoiceState
  inventory: EditableInventoryState
  notes: EditableNotes
}
```

### 8.1 `meta`

```ts
type EditableMeta = {
  characterId: string
  createdAt: string
  updatedAt: string
  status: 'draft' | 'completed'
}
```

### 8.2 `wizard`

```ts
type EditableWizardState = {
  currentStep: number
  completedSteps: number[]
  pendingChoiceIds: string[]
}
```

说明：

- `pendingChoiceIds` 用于恢复未处理完的选择链

### 8.3 `identity`

```ts
type EditableIdentity = {
  name?: string
  playerName?: string
  portraitUrl?: string
  alignment?: string
  faith?: string
  homeland?: string
  age?: string
  height?: string
  weightText?: string
  gender?: string
  appearance?: string
  personalityTraits?: string[]
  ideals?: string[]
  bonds?: string[]
  flaws?: string[]
  backstory?: string
}
```

### 8.4 `build`

```ts
type EditableBuildState = {
  level: number
  classEntries: EditableClassSelection[]
  selectedSpeciesId?: string
  selectedSubraceId?: string
  selectedBackgroundId?: string
  abilityGenerationMode?: 'standard_array' | 'point_buy' | 'rolled' | 'manual'
  baseAbilityScores: Partial<AbilityScoreMap>
}

type EditableClassSelection = {
  classId: string
  level: number
  subclassId?: string
}
```

### 8.5 `choices`

```ts
type EditableChoiceState = {
  selectedFeatIds: string[]
  selectedSpellIds: string[]
  selectedSkillIds: string[]
  selectedToolIds: string[]
  selectedLanguageIds: string[]
  resolvedChoices: Record<string, EditableResolvedChoice>
}

type EditableResolvedChoice = {
  choiceId: string
  selectedOptionIds: string[]
  selectedEntityRefs?: EntityReference[]
  resolvedAt: string
}
```

说明：

- 运行时的待选池由计算引擎根据规则生成
- 编辑态只保存“用户已经对哪些 choice 做出了什么选择”

### 8.6 `inventory`

```ts
type EditableInventoryState = {
  selectedStartingPackageIds: string[]
  manualItems: InventoryLineInput[]
  coins: CurrencyMap
  equippedItemIds: string[]
}

type InventoryLineInput = {
  itemId: string
  quantity: number
  source?: string
}
```

### 8.7 `notes`

```ts
type EditableNotes = {
  customNotes?: string[]
}
```

## 9. 计算态角色对象建议结构

计算态角色对象应直接对应 [最终角色状态结构.md](E:/YJF/DND2024characterbuilder-2.0/docs/03_规则与状态/最终角色状态结构.md)。

建议结构：

```ts
type ComputedCharacterState = FinalCharacterState
```

也就是至少包含：

- `meta`
- `identity`
- `progression`
- `abilities`
- `combat`
- `proficiencies`
- `traits`
- `spells`
- `inventory`
- `carrying`
- `notes`
- `sources`
- `computed`

## 10. 二者的关键区别

### 10.1 编辑态保存“为什么”

它保存的是：

- 用户做了什么选择
- 这些选择还缺什么
- 当前流程进行到哪

### 10.2 计算态保存“结果是什么”

它保存的是：

- 最终结算值
- 最终展示内容
- 最终汇总结构

### 10.3 一个例子

编辑态可能是：

- `classEntries = [{ classId: 'barbarian-xphb', level: 1 }]`
- `selectedBackgroundId = 'acolyte-xphb'`
- `selectedFeatIds = ['magic-initiate-xphb']`
- `pendingChoiceIds = ['magic-initiate-spell-pool']`

计算态则是：

- `armorClass = 15`
- `skillProficiencies = ['洞悉', '宗教', '运动']`
- `knownSpells = ['神导术', '圣火术', '治疗创伤']`
- `carryCapacityLb = 240`

## 11. 角色创建完成后重新编辑的正式结论

本规范的核心结论之一是：

**角色创建完成后，必须仍然可以重新自定义编辑角色。**

为保证这一点，角色存档必须至少保存：

- 一份完整的 `editableState`
- 一份当前最新的 `computedState`

重新编辑时的规则：

- 进入创建流程时，以 `editableState` 作为恢复源
- 打开角色卡时，以 `computedState` 作为展示源
- 用户修改任一编辑项后，重新从 `editableState` 计算新的 `computedState`

因此：

- `computedState` 不能替代 `editableState`
- `editableState` 也不能代替 `computedState`

---

## 12. 保存与加载规范

建议角色存档采用如下结构：

```ts
type CharacterSaveFile = {
  editableState: EditableCharacterState
  computedState: ComputedCharacterState
}
```

### 12.1 保存规则

保存时：

1. 先保存当前 `editableState`
2. 再基于它生成并保存最新 `computedState`

### 12.2 加载规则

加载时：

- 如果进入编辑流程，优先读 `editableState`
- 如果进入展示页面，优先读 `computedState`

### 12.3 重算规则

若检测到以下情况，应强制重算：

- 规则版本变化
- 数据库版本变化
- `editableState.updatedAt` 晚于 `computedState.updatedAt`
- 某些条目的 source 版本已切换

---

## 13. 统一数据流

建议统一采用以下数据流：

```text
用户选择
  -> 更新 editableState
  -> 触发统一计算
  -> 生成 computedState
  -> 更新角色卡展示
```

重新编辑时：

```text
读取 editableState
  -> 恢复向导步骤与待选池
  -> 用户修改
  -> 重新生成 computedState
```

---

## 14. 哪些字段必须只存在编辑态

以下字段原则上只应存在编辑态：

- `currentStep`
- `completedSteps`
- `pendingChoiceIds`
- 原始未确认选择
- 用户还没完成的能力分配

---

## 15. 哪些字段必须只存在计算态

以下字段原则上只应存在计算态：

- `armorClass`
- `armorClassBreakdown`
- `initiative`
- `passivePerception`
- 最终熟练项总表
- 最终法术总表
- 最终负重状态
- 最终财富汇总
- 最终来源追踪汇总

---

## 16. 哪些字段可以两边都有

以下字段允许编辑态和计算态都保留，但语义不同：

- `name`
- `level`
- `selectedFeatIds` / `featIds`
- `selectedSpellIds` / `knownSpells`
- `currency`
- `equippedArmorId`

区别在于：

- 编辑态保存“选择输入”
- 计算态保存“最终结果”

---

## 17. 实现建议

建议代码层明确拆成三个模块：

### 17.1 编辑状态模块

职责：

- 更新用户选择
- 维护向导步骤
- 维护待选池

### 17.2 统一计算模块

职责：

- 接收 `editableState`
- 输出 `computedState`

### 17.3 展示模块

职责：

- 角色卡优先读取 `computedState`
- 编辑器优先读取 `editableState`

---

## 18. 最终结论

编辑态与计算态的区别可以概括为：

- 编辑态是“角色怎么被做出来”
- 计算态是“角色最终是什么样”

为了实现“创建完毕后仍可重新自定义编辑角色”这一基础功能，必须同时保存二者。  
只保存计算态，会失去可编辑性；只保存编辑态，会缺少稳定展示和快速结算结果。
