# 规则计算引擎设计

## 1. 文档目标

本文件用于把已有的规则口径、类型契约和分步建卡要求落成工程可执行的计算引擎设计。

它回答的问题是：

- 规则计算从哪里开始
- 计算过程拆成哪些阶段
- 固定授予和选择授予如何处理
- 嵌套授予如何递归解析
- 冲突和重复如何处理
- 在什么时机触发重算

## 2. 设计目标

规则计算引擎必须同时满足：

1. 能从 `EditableCharacterState` 稳定生成 `ComputedCharacterState`
2. 能在上游选择变化后局部重算或整体重算
3. 能产出待选项池，而不是只产出最终结果
4. 能记录结果来源链
5. 不依赖 UI 组件里的硬编码规则

---

## 3. 核心输入与输出

### 3.1 输入

```ts
type ComputeCharacterInput = {
  editableState: EditableCharacterState
  repository: DomainRepository
}
```

其中 `repository` 提供运行时领域实体查询能力：

```ts
type DomainRepository = {
  getClass(id: EntityId): ClassEntity | undefined
  getSubclass(id: EntityId): SubclassEntity | undefined
  getSpecies(id: EntityId): SpeciesEntity | undefined
  getBackground(id: EntityId): BackgroundEntity | undefined
  getFeat(id: EntityId): FeatEntity | undefined
  getSpell(id: EntityId): SpellEntity | undefined
  getItem(id: EntityId): ItemEntity | undefined
}
```

### 3.2 输出

```ts
type ComputeCharacterResult = {
  computedState: ComputedCharacterState
  runtime: ComputeRuntimeState
}
```

说明：

- `computedState` 是最终可展示、可存档的角色快照
- `runtime` 是引擎内部调试与待选项生成结果，不直接暴露给最终角色卡

---

## 4. 引擎分层

建议把引擎拆成 4 层：

1. 选择解析层
2. 授予展开层
3. 规则归并层
4. 最终映射层

统一流程：

`editableState -> normalize selections -> expand grants -> aggregate effects -> build computedState`

---

## 5. 运行时状态设计

运行时状态是引擎中间层，不对最终 UI 直接暴露，但必须可调试。

```ts
type ComputeRuntimeState = {
  selectedEntities: SelectedEntitySet
  pendingChoices: RuntimePendingChoice[]
  appliedEffects: AppliedEffectRecord[]
  grantedEntities: GrantedEntityRecord[]
  sourceIndex: RuntimeSourceIndex
  warnings: RuntimeWarning[]
}
```

### 5.1 已选实体集合

```ts
type SelectedEntitySet = {
  classes: ClassEntity[]
  subclasses: SubclassEntity[]
  species?: SpeciesEntity
  background?: BackgroundEntity
  explicitFeats: FeatEntity[]
  explicitSpells: SpellEntity[]
  items: ItemEntity[]
}
```

### 5.2 待选项

```ts
type RuntimePendingChoice = {
  choiceId: string
  sourceId: string
  sourceType: DomainEntityType | 'runtime'
  prompt: string
  count: number
  status: 'pending' | 'resolved' | 'invalid'
  options: RuntimeChoiceOption[]
}

type RuntimeChoiceOption = {
  optionId: string
  label: string
  entityRefs?: EntityReference[]
  effects?: RuleEffect[]
}
```

### 5.3 已应用效果

```ts
type AppliedEffectRecord = {
  effectId: string
  sourceId: string
  sourceType: DomainEntityType | 'runtime'
  effect: RuleEffect
  applied: boolean
  skippedReason?: string
}
```

### 5.4 已授予实体

```ts
type GrantedEntityRecord = {
  entityType: DomainEntityType
  entityId: EntityId
  sourceId: string
  sourceType: DomainEntityType | 'runtime'
  grantPath: string[]
}
```

### 5.5 来源索引

```ts
type RuntimeSourceIndex = Record<string, RuntimeSourceTrace[]>

type RuntimeSourceTrace = {
  resultKey: string
  sourceId: string
  sourceType: DomainEntityType | 'runtime'
  note?: string
}
```

## 6. 计算阶段设计

引擎建议严格按阶段运行，不允许在 UI 中穿插私有规则。

### 阶段 1：选择归一化

目标：

- 从 `editableState` 中提取已选职业、背景、种族、显式专长、显式法术、装备
- 把松散输入整理成统一 `SelectedEntitySet`

输出：

- `selectedEntities`
- 初始 `warnings`

关键规则：

- 任何缺失实体引用都必须进入 warning
- 不允许静默忽略找不到的 id

### 阶段 2：基础授予收集

目标：

- 收集职业、背景、种族、显式专长、显式装备上的 `effects` 和 `grants`
- 收集职业等级条目的 `effects` 和 `grants`

输出：

- 初始 grant 队列
- 初始 effect 队列

关键规则：

- 职业等级相关授予只处理当前等级范围内的条目
- 子职业授予只有在满足触发条件时才进入队列

### 阶段 3：递归授予展开

目标：

- 递归处理 `RuleGrant`
- 产出新增实体、待选项和效果

输出：

- `grantedEntities`
- `pendingChoices`
- 补充后的 `appliedEffects`

统一策略：

1. 遇到 `fixed` 授予，直接展开到 effect/entity 队列
2. 遇到 `choice` 授予：
   - 若 `editableState` 已有对应选择，则展开其选中项
   - 若没有选择，则进入 `pendingChoices`
3. 新授予的 feat/spell/item 若自身仍带有 `effects` 或 `grants`，继续递归

终止条件：

- 同一路径中的同一实体重复出现时，停止继续递归，写入 warning，避免死循环

### 阶段 4：效果归并

目标：

- 将所有 effect 统一归并到中间结果模型

输出：

- 中间聚合结果 `AggregateState`

建议按以下子域聚合：

- 身份与构筑
- 属性
- 熟练
- 法术
- 战斗
- 装备
- 负重
- 来源链

### 阶段 5：冲突处理

目标：

- 对同类效果进行去重、互斥选择和优先级处理

输出：

- 冲突处理后的稳定聚合结果

### 阶段 6：最终映射

目标：

- 把聚合结果映射为 `ComputedCharacterState`

输出：

- `computedState`

## 7. 中间聚合结果模型

建议引擎内部先写入统一聚合结构，再映射到最终角色结构。

```ts
type AggregateState = {
  identity: AggregateIdentity
  progression: AggregateProgression
  abilities: AggregateAbilities
  proficiencies: AggregateProficiencies
  spells: AggregateSpells
  combat: AggregateCombat
  inventory: AggregateInventory
  carrying: AggregateCarrying
  traits: AggregateTraits
  sources: RuntimeSourceIndex
}
```

这样做的原因：

- 便于在规则阶段做冲突处理
- 便于后续扩展兼职、多来源法术和复杂装备逻辑
- 避免一边算一边直接修改最终角色结构，导致难以调试

## 8. 效果处理规则

### 8.1 属性效果

处理目标：

- 属性固定提升
- 属性多选提升
- 属性上限影响

统一策略：

- 先写入原始加值列表
- 再按顺序合并到最终属性值
- 同时写入来源追踪

### 8.2 熟练效果

处理目标：

- 豁免熟练
- 技能熟练
- 工具熟练
- 武器熟练
- 护甲训练

统一策略：

- 并集去重
- 保留多来源记录
- 若同一来源要求重选，则放回待选项池

### 8.3 专长与法术效果

处理目标：

- 授予专长
- 授予法术
- 授予施法能力
- 授予可选法术池

统一策略：

- 先记录“显式选择获得”与“规则授予获得”两个来源维度
- 同名同 id 去重
- 不同来源共享同一法术时，来源链累加

### 8.4 AC 公式效果

处理目标：

- 基础无甲公式
- 护甲提供的 AC 公式
- 特性提供的替代公式
- 盾牌加值

统一策略：

1. 收集所有可用 AC 公式
2. 计算每个公式的结果值
3. 对互斥公式取合法最优结果
4. 盾牌类附加值按兼容规则叠加
5. 把最终选中的公式写入 breakdown

### 8.5 资源与装备效果

处理目标：

- 起始装备
- 金币
- 负重
- 可装备物品

统一策略：

- 所有装备先进入清单
- 再按已装备状态参与 AC / 攻击 / 负重计算
- 钱币重量独立统计

## 9. 冲突处理策略

### 9.1 去重型冲突

适用于：

- 熟练
- 法术
- 专长
- 语言

规则：

- 结果层去重
- 来源层保留全部来源

### 9.2 互斥型冲突

适用于：

- AC 公式
- 装备方案 A/B
- 某些二选一特性

规则：

- 若用户已明确选择，以用户选择为准
- 若未选择但规则要求必须选，进入 pending
- 若系统能自动选择最优但规则不允许自动决定，则不得代替用户决策

### 9.3 非法型冲突

适用于：

- 选择数量超限
- 选择了不在池中的条目
- 授予链出现循环
- 缺失实体引用

规则：

- 写入 warning
- 标记对应 choice 为 invalid
- 阻止角色进入 playable 状态

## 10. 待选项生成机制

待选项不应由 UI 自己拼，而应由计算引擎在运行时生成。

### 10.1 待选项来源

- 背景授予技能选择
- 背景授予起源专长选择
- 种族授予语言选择
- 职业授予技能选择
- 专长授予法术或属性选择
- 装备方案选择

### 10.2 待选项生成规则

若某个 `ChoiceGrant` 满足以下条件，则进入 pending：

1. 当前计算路径已触发该 grant
2. 当前 `editableState.choices.resolvedChoices` 中没有合法选择结果

### 10.3 待选项关闭规则

满足以下条件时，待选项从 pending 转为 resolved：

1. 选择数量满足规则
2. 所选项都在合法选项池内
3. 没有触发重复选择或无效选择

## 11. 重算时机

### 11.1 必须触发整体重算的变更

- 职业变化
- 子职业变化
- 背景变化
- 种族变化
- 属性分配变化
- 专长选择变化
- 法术选择变化
- 装备方案变化

### 11.2 可以接受整体重算的阶段

MVP 阶段不必做细粒度增量重算。  
建议策略：

- 任意关键输入变化后执行一次全量重算

理由：

- 当前规则链路复杂，先保证正确性
- 后续性能瓶颈出现后再考虑局部失效机制

### 11.3 重算触发点

建议在以下时机触发：

- 用户完成某一步并点击下一步
- 用户在步骤内完成一项关键选择
- 用户从角色详情重新进入编辑并提交修改

---

## 12. 错误与警告策略

### 12.1 warning

适用于：

- 文本化效果尚未结构化
- 非阻断性重复来源
- 可降级处理的兼容问题

### 12.2 blocking error

适用于：

- 缺失关键实体
- 关键必选项未完成
- 循环授予未被打断
- 非法选择导致无法计算最终结果

### 12.3 状态影响

- 有 warning：允许保存 draft
- 有 blocking error：不允许保存 playable

## 13. MVP 实现建议

MVP 不要一次做通用规则 DSL。建议先实现以下固定骨架：

1. 基于 `RuleEffect` / `RuleGrant` 的解释器
2. 基于队列的递归 grant 展开器
3. 基于 `AggregateState` 的归并器
4. 基于 `FinalCharacterState` 的映射器

建议代码结构：

```ts
src/domain/engine/compute-character.ts
src/domain/engine/normalize-selections.ts
src/domain/engine/expand-grants.ts
src/domain/engine/apply-effects.ts
src/domain/engine/resolve-conflicts.ts
src/domain/engine/build-computed-state.ts
```

## 14. 最小验收标准

引擎完成的判断标准：

1. 给定一份合法 `EditableCharacterState`，能稳定输出 `ComputedCharacterState`
2. 给定一份缺少关键选择的 `EditableCharacterState`，能稳定输出 `pendingChoices`
3. 修改上游选择后，结果能整体重算且不会残留旧来源
4. 同一法术、熟练、专长来自多来源时，结果不重复但来源链保留
5. AC、负重和法术授予链能给出可解释的 breakdown
